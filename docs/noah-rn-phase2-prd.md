## 0. Architectural Invariants

These are non-negotiable constraints derived from 15 distilled research reports and board direction. Every implementation decision in this PRD must satisfy all invariants simultaneously. Violations require board escalation — never silent resolution.

```yaml
invariants:
  regulatory_posture:
    classification: "HITL Category II — documentation assistance"
    cds_exemption_test:
      - "no_image_analysis: true"
      - "displays_analyzes_medical_info: true"
      - "supports_without_replacing_judgment: true"
      - "enables_independent_review: true"
    constraint: "No skill may produce autonomous clinical decisions without human review. Drafts only. The moment this invariant breaks, noah-rn enters FDA device jurisdiction."
    philosophy: "compliance-ready, not compliance-driven — guidelines are clinical knowledge inputs, not architectural pillars"

  safety_hierarchy:
    layer_0: "Deterministic plausibility checks (non-LLM, hard-coded, unprompt-injectable)"
    layer_1: "Input validation and sanitization at every boundary"
    layer_2: "Skill-level completeness checklists (omission > commission)"
    layer_3: "Confidence calibration in output"
    layer_4: "Human review (current boundary, not permanent architecture — design for Jethro-principle delegation)"

  output_format:
    spec: "four-layer explanation"
    layers:
      - "summary: actionable clinical synthesis"
      - "evidence: supporting data with linked references"
      - "confidence: calibrated uncertainty indicators"
      - "provenance: source attribution with version metadata"
    footer: "Generated by noah-rn v{version} / {skill_name} v{skill_version} / {knowledge_source} v{knowledge_version}"

  provenance:
    scope: "system-wide — every file, skill, tool, agent card, knowledge artifact"
    required_fields: ["source", "version", "date", "last_verified"]
    rationale: "agents consume everything; everything must be machine-describable"

  model_agnosticism:
    constraint: "No skill, hook, or agent may contain provider-specific API calls. All model interaction routes through a single centralized interface."
    rationale: "The harness > the model. Retrieval quality > model scale."

  a2a_readiness:
    constraint: "Every skill must be describable by a machine-readable Agent Card even if no external agent currently consumes it."
    rationale: "A2A is core to all missions, not deferred."

  thoroughness_over_speed:
    constraint: "When the routing agent faces ambiguity between faster/shallow and slower/thorough, default to thorough."
    rationale: "The cost of a missed finding always exceeds the cost of a 500ms delay."
```

## 2. Phase 2 Deliverables

### 2.1 Agent Routing (`plugin/agents/`)

**Purpose**: Receive clinical intent from the user, classify it, select the appropriate skill(s), invoke them with scoped context, validate outputs, and compose the final response in four-layer explanation format.

**Architecture**:

```
user_input
  → input_sanitizer (hooks/tier-1)
  → intent_classifier (agents/router)
      → complexity_assessment → assigns model_tier hint
      → skill_selection → one or more skills
      → context_scoping → minimum necessary data per skill (least-privilege)
  → skill_invocation (sequential or parallel based on dependency)
  → output_validator (hooks/tier-1 + tier-2)
  → completeness_checker (hooks/skill-specific checklist)
  → confidence_scorer
  → response_composer (four-layer format + footer)
  → user
```

**Interface contract for the router**:

```yaml
input:
  raw_text: string          # user's natural language input
  session_context: object   # accumulated encounter state (working memory)

output:
  response:
    summary: string         # actionable clinical synthesis
    evidence: array         # [{claim: string, source: string, source_version: string}]
    confidence:
      overall: float        # 0.0-1.0
      flags: array          # sections requiring double-check
    provenance:
      skills_invoked: array # [{skill_name, skill_version, knowledge_sources: [{name, version, date}]}]
      model: string         # model identifier used (opaque — no provider coupling)
      timestamp: ISO8601
  routing_metadata:
    intent_classified: string
    complexity_tier: enum [simple, moderate, complex]
    skills_considered: array
    skills_selected: array
    escalation_reason: string | null  # why thorough > fast, if applicable
```

**Routing logic constraints**:
- If ambiguous between two skills, invoke both and present consolidated output (thoroughness > speed)
- If no skill matches intent, respond with "this is outside my current capabilities" with the nearest-match skill suggestion — never hallucinate a response from an unmatched skill
- If a skill's required clinical context is missing from input, request it explicitly before proceeding — never infer missing vitals, labs, or assessments
- Cross-skill awareness: if skill output contains findings that should trigger another skill (e.g., assessment reveals HR <60 → suggest bradycardia protocol), surface this as a suggestion in the response, not an autonomous invocation

**Agent Card schema** (A2A readiness):

```json
{
  "$schema": "https://noah-rn.dev/agent-card/v1",
  "name": "noah-rn-router",
  "version": "2.0.0",
  "description": "Clinical nursing documentation assistant — routes intent to specialized skills",
  "capabilities": ["nursing_documentation", "clinical_calculators", "protocol_reference", "medication_lookup"],
  "input_modalities": ["text"],
  "output_modalities": ["text"],
  "hitl_category": "II",
  "limitations": ["adult_patients_only", "no_image_analysis", "no_autonomous_prescribing"],
  "supported_skills": []  // populated dynamically from skill registry
}
```

### 2.2 Hooks Foundation (`plugin/hooks/`)

**Purpose**: Tiered guardrail system operating at input, intra-processing, and output boundaries. Non-LLM deterministic checks form the irreducible safety floor.

**Tier architecture**:

```yaml
tier_1_deterministic:
  latency_budget: "~0ms"
  implementation: "shell/jq — no model calls"
  triggers: "every input and every output"
  checks:
    dosage_range:
      description: "Validate medication dosages against pharmacological bounds"
      implementation: "lookup against knowledge/drug-ranges.json"
      action_on_fail: "BLOCK output, append warning, flag for review"
    unit_verification:
      description: "Detect unit mismatches (mg vs mcg, mL vs L, kg vs lbs)"
      implementation: "regex + unit conversion validation"
      action_on_fail: "BLOCK output, request clarification"
    negation_integrity:
      description: "Detect missing or inverted negation in critical phrases"
      patterns: ["do not resuscitate", "no known allergies", "hold medication"]
      implementation: "pattern matching against known critical-negation phrases"
      action_on_fail: "FLAG in confidence layer"
    input_sanitization:
      description: "Strip instruction-like patterns from clinical input"
      implementation: "regex filter for prompt-injection signatures"
      action_on_fail: "SANITIZE and log, continue with cleaned input"

tier_2_classifier:
  latency_budget: "10-50ms"
  implementation: "lightweight heuristic or small classifier"
  triggers: "post-routing, pre-skill-invocation"
  checks:
    scope_validation:
      description: "Is this query within the invoked skill's clinical domain?"
      implementation: "keyword/heuristic match against skill.limitations and skill.scope"
      action_on_fail: "REROUTE to correct skill via router"
    context_completeness:
      description: "Does the input contain minimum required clinical data for this skill?"
      implementation: "skill-specific required_fields check"
      action_on_fail: "REQUEST missing data before proceeding"

tier_3_llm_judge:
  latency_budget: "300-800ms"
  implementation: "model call through centralized interface"
  triggers: "post-generation, high-complexity outputs only"
  status: "INTERFACE DEFINED, IMPLEMENTATION DEFERRED"
  checks:
    clinical_plausibility:
      description: "Does this output make clinical sense given the input context?"
    contradiction_detection:
      description: "Does this output contradict information in the input context?"
    guideline_conflict_surfacing:
      description: "Are there conflicting guideline recommendations? Surface both."
```

**Hook execution model**: Hooks execute as a pipeline. Tier 1 runs on every invocation. Tier 2 runs post-routing. Tier 3 runs only when `complexity_tier == complex`. Any hook can BLOCK (halt output), FLAG (annotate confidence layer), SANITIZE (clean input), REROUTE (redirect to different skill), or REQUEST (ask for missing data).

### 2.3 Knowledge Architecture (`knowledge/`)

**Purpose**: Curated clinical reference data with provenance metadata, freshness tracking, and a retrieval interface abstracted for future RAG migration.

**Provenance header spec** (every file in `knowledge/`):

```yaml
---
source: "AHA/ACLS 2025 Guidelines"
source_url: "https://..."
version: "2025.1"
date: "2025-10-15"
evidence_grade: "Class I / Level A"  # where applicable
last_verified: "2026-03-28"
next_review: "2026-06-28"
scope: "adult_cardiac_arrest"
limitations: ["not_applicable_to_pediatrics", "not_applicable_to_neonates"]
---
```

**FRESHNESS.md manifest**:

```markdown
# Knowledge Freshness Manifest

| File | Source | Version Date | Last Verified | Next Review | Status |
|------|--------|-------------|---------------|-------------|--------|
| acls-2025.md | AHA | 2025-10-15 | 2026-03-28 | 2026-06-28 | CURRENT |
| sepsis-bundle.md | SSC | 2024-01-01 | 2026-03-28 | 2026-06-28 | CURRENT |
...
```

**Automated freshness checking**: Implement a scheduled task (cron, CI action, or agentic harness) that:
1. Parses `FRESHNESS.md` for `next_review` dates
2. Flags any file past its review date as `STALE`
3. Outputs a report for board review — does not auto-update content
4. Optionally checks source URLs for HTTP 200 / last-modified headers

**Retrieval interface abstraction**:

```
knowledge.retrieve(query: string, scope: string) → KnowledgeResult[]

interface KnowledgeResult {
  content: string
  source: string
  version: string
  date: string
  evidence_grade: string | null
  relevance_score: float  // 0.0-1.0, currently 1.0 for all (context stuffing)
}
```

Current implementation: file-based context stuffing (load relevant knowledge files into skill context based on scope tag matching). The interface is designed so the implementation can swap to embedding-based retrieval without changing the skill contract.

### 2.4 Output Standardization

**Every skill output must conform to the four-layer explanation format.** Implement as a shared output composer invoked after skill execution and hook validation.

```
compose_output(skill_result, hook_annotations, routing_metadata) → StandardOutput

interface StandardOutput {
  summary: string
  evidence: Evidence[]
  confidence: {
    overall: float
    flags: ConfidenceFlag[]
  }
  provenance: {
    noah_rn_version: string
    skill: { name: string, version: string }
    knowledge_sources: { name: string, version: string, date: string }[]
    model: string  // opaque identifier
    timestamp: string
  }
  footer: string  // human-readable attribution line
}

interface Evidence {
  claim: string
  source: string
  source_version: string
}

interface ConfidenceFlag {
  section: string
  reason: string
  severity: enum [info, check, warning]
}
```

**Completeness checklists** (skill-specific, defined per skill):

```yaml
# Example: shift-report skill
completeness_checklist:
  required_systems:
    - neurological
    - cardiovascular
    - respiratory
    - gastrointestinal
    - genitourinary
    - integumentary
    - musculoskeletal
    - endocrine
    - hematologic
    - immunologic
    - pain
    - psychosocial
  required_sections:
    - lines_and_access
    - medications_due
    - allergies
    - code_status
    - isolation_precautions
  on_missing: "FLAG as confidence.warning with specific missing section identified"
```

### 2.5 Clinical Calculators (`plugin/skills/calculators/`)

**Purpose**: Deterministic clinical scoring tools. These are NOT LLM-dependent — they are pure computation with structured input and validated output.

**Target calculators (Phase 2)**:

| Calculator | Inputs | Output | Clinical Context |
|-----------|--------|--------|-----------------|
| GCS | eye (1-4), verbal (1-5), motor (1-6) | total (3-15), severity classification | trauma, neuro assessment |
| NIHSS | 11 scored domains | total (0-42), stroke severity | stroke assessment |
| APACHE II | age, temp, MAP, HR, RR, oxygenation, pH, Na, K, creatinine, Hct, WBC, GCS, chronic health | score (0-71), predicted mortality % | ICU acuity |
| Wells (PE) | clinical signs DVT, PE likely, HR>100, immobilization, previous VTE, hemoptysis, malignancy | score, risk category | PE workup |
| Wells (DVT) | active cancer, paralysis, bedridden, tenderness, swelling, pitting edema, collateral veins, previous DVT, alternative dx | score, risk category | DVT workup |
| CURB-65 | confusion, urea, RR, BP, age≥65 | score (0-5), disposition recommendation | pneumonia disposition |
| Braden | sensory perception, moisture, activity, mobility, nutrition, friction/shear | score (6-23), risk category | pressure injury risk |
| RASS | observation-based | score (-5 to +4), sedation level | sedation assessment |
| CPOT | facial expression, body movements, muscle tension, ventilator compliance/vocalization | score (0-8), pain assessment | non-verbal pain |

**Implementation constraints**:
- Pure deterministic functions — no model calls
- Input validation: reject out-of-range values, require all fields
- Output includes: raw score, interpretation, clinical significance, and the formula/reference used
- Each calculator includes a provenance header citing the original validation study
- Cross-skill awareness: calculator results feed back to the router as potential triggers (e.g., GCS ≤ 8 → suggest airway management protocol; NIHSS > 0 → suggest stroke skill)

### 2.6 Unit Conversion Tool (`tools/unit-convert`)

**Purpose**: Weight-based dosing calculations, drip rate computations, I&O totals. Deterministic, no model dependency.

**Core functions**:

```
weight_based_dose(weight_kg: float, dose_per_kg: float, unit: string) → {dose: float, unit: string}
drip_rate(dose: float, concentration: float, weight_kg: float | null) → {rate_ml_hr: float, gtts_min: float}
io_total(inputs: {volume: float, type: string}[], outputs: {volume: float, type: string}[]) → {total_in: float, total_out: float, net: float}
convert(value: float, from_unit: string, to_unit: string) → {result: float, unit: string}
```

**Safety checks (Tier 1 hooks)**:
- Reject physiologically impossible weights (<0.5 kg, >500 kg)
- Flag extreme dosage results (>10x typical adult dose) as `confidence.warning`
- Validate unit compatibility (cannot convert mg to mL without concentration)

## 6. Reference Architecture Constraints

For agents implementing against this PRD, these are the binding constraints from the research distillation that govern HOW you build, not just WHAT you build.

```yaml
constraints:
  knowledge:
    - "Only ingest authoritative source publications into knowledge/ — never derivative clinical notes"
    - "Drug interaction checking requires cross-reference of minimum 2 independent sources"
    - "When guideline conflict is detected, surface BOTH positions with sources — never silently resolve"
    - "knowledge/ files must break the copy-forward cycle — require current clinical context, not templates"

  safety:
    - "Deterministic checks are the irreducible safety floor — they cannot depend on model availability"
    - "Assume adversarial input at every boundary (94.4% prompt injection success rate in clinical LLMs)"
    - "Omission is more dangerous than commission — completeness checklists are safety architecture"
    - "Confidence calibration is a safety feature, not UX — suppresses automation bias"

  design:
    - "Design interfaces for the next tier, implement for the current one"
    - "10-20-70 allocation: 10% model sophistication, 20% harness engineering, 70% workflow fit"
    - "Weakest-link governs system maturity — don't advance skills past monitoring/testing capability"
    - "Automate the watching, humanize the deciding"

  identity:
    - "antrhodiscernment: clinical judgment (contradiction detection, evidence weighting, pattern recognition) encoded as system architecture is the competitive moat"
    - "This project is built by a critical care nurse — the harness encodes ICU-grade clinical hierarchy"
    - "No engineer without bedside experience can correctly weight these evidence hierarchies — that's the point"
```
