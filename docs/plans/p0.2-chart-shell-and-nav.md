# P0.2 — Chart shell and nav (nursing-station)

## Context

P0.1 introduces `PatientChartLayout` as a route wrapper for `/Patient/:id`, hosts the persistent header, and encodes the active chart section as a `?tab=` URL parameter. That unblocks the header contract, but leaves the intra-chart navigation as a top tab strip — the roadmap calls this out as "too demo-like" and specifies P0.2 as the follow-on that "replaces the demo-like top-tab feel with a stable clinician shell." This plan promotes the tab strip to a left rail, converts `?tab=` to real nested routes, and makes Overview the default landing view. It depends on P0.1 landing first.

Feature-matrix row: `Chart shell / nav | P0 | nursing-station | route stability; no context loss; screenshot each nav state`.

Source inspirations (from `docs/analysis/ui-resource-ledger.md`): Epic chart review shell (persistent header + left navigator), OpenEMR Dashboard Context Manager (context-aware widget sets), `medplum-provider` repo (shell composition, routing, patient-summary integration).

## Verified environment

- React Router 7.14 nested routing with `<Outlet />` already in use after P0.1.
- Mantine 8.3 `AppShell` currently owns the app-level sidebar (PATIENTS | TASKS at 220px). The chart-level navigator will be a **second, inner** rail hosted by `PatientChartLayout` — not a mutation of `AppShell.Navbar`.
- No deep-link telemetry to preserve; `?tab=` parameter introduced in P0.1 can be replaced without deprecation window.

## Architectural decisions

1. **Nested routes, not URL parameters.** Convert the five chart sections to real routes: `/Patient/:id/overview`, `/timeline`, `/vitals`, `/labs`, `/meds`, `/tasks`. The `?tab=` parameter from P0.1 is retired. Rationale: nested routes give each section its own component identity, enable per-section code-splitting later, let browser back/forward behave correctly, and match the Medplum-provider shell pattern. Acceptance check "no context loss" becomes trivially true — each section is just an `<Outlet />` swap under the layout.
2. **Left rail inside the chart, not above it.** A 200px vertical navigator lives inside `PatientChartLayout`, **below** the persistent patient header and **to the right of** the app-level sidebar. The outer AppShell sidebar keeps PATIENTS / TASKS (app-level nav); the inner rail holds the six chart sections (chart-level nav). This matches the Epic chart review shell convention of keeping workflow-level and patient-level navigation visually separated.
3. **Overview is the default landing route.** A React Router index route redirects `/Patient/:id` → `/Patient/:id/overview`. The Overview page itself is scaffolded as an empty placeholder in this PR (filled by P0.3). No redirect logic outside the route config.
4. **Rail items are `NavLink` components with `end` prop on overview.** Active state uses React Router's built-in `isActive`; no local state. Each item gets `data-testid="chart-nav-{section}"`.
5. **Section components are lazy-loaded** via `React.lazy` + `<Suspense fallback={null}>` so one slow section doesn't block others. The fallback is deliberately null (not a spinner) because the persistent header + left rail remain rendered — the user always has context.
6. **Existing `PatientChartPage.tsx` is retired.** After P0.1 it is already reduced to a tab-content switch. P0.2 replaces that switch with per-route components (`OverviewPage`, `TimelinePage`, `VitalsPage`, `LabsPage`, `MedsPage`, `PatientTasksPage`), one per section. The existing tab-content bodies move into these new files 1:1.

## Acceptance-criteria mapping

| Feature-matrix check | How this plan satisfies it |
|---|---|
| route stability | each section is a distinct React Router route; back/forward and deep links work natively |
| no context loss | layout does not remount on section change; `<Outlet />` is the only thing swapping |
| screenshot each nav state | six stable URLs + six `data-testid="chart-nav-{section}"` rail items = six distinct screenshot targets for Phase C |
