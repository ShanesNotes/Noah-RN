# P0.3 — Overview page (nursing-station)

## Context

P0.2 scaffolds `OverviewPage` as an empty placeholder at `/Patient/:id/overview` and makes it the default landing route. This plan populates it. The roadmap frames Overview as "the default chart entry that answers what matters now" — the clinician's first glance at a patient. It is a composed summary panel, not a new workflow surface. No inline edits, no agent actions; those belong to later phases.

> **Superseded in part (2026-04-14):** Decision #4 ("No cross-card state — cards do not share fetches") is reversed by `docs/plans/p0-optimization-pass.md`, which introduces a shared `usePatientChartData` hook batched via Medplum's `autoBatchTime`. Per-card *loading* isolation is preserved via skeletons; only the *fetch* layer is collapsed. Read that plan alongside this one when picking up B4 work.

Feature-matrix row: `Overview page | P0 | nursing-station | all required summary blocks visible; empty-state sane; screenshot baseline`.

Source inspirations (from `docs/analysis/ui-resource-ledger.md`): Epic summary / SBAR patterns, Medplum summary concepts, OpenEMR Clinical Decision Rules (for missing-data indicators).

## Verified environment

- P0.2 delivers: route `/Patient/:id/overview`, `OverviewPage.tsx` stub consuming `patient` via `usePatientFromOutlet()`, lazy-loaded under `<Suspense fallback={null}>`.
- Medplum hooks available: `useResource`, `useSearch`, `useSearchOne`, `useSearchResources`.
- Existing clinical-mcp gap-detection logic lives in `services/clinical-mcp/` and exposes a context-bundle shape; **this plan does not call clinical-mcp** — missing-data indicators on Overview are derived locally from FHIR reads (empty result → "missing data" badge). Deeper integration with clinical-mcp gap logic is P1 (`Missing-data / gaps indicators` matrix row, which already says "wrap existing gap logic").

## Architectural decisions

1. **Card grid, not a single dense document.** Eight independent `OverviewCard` components arranged in a responsive Mantine `SimpleGrid` (3-col ≥1440px, 2-col ≥960px, 1-col mobile). Each card is self-contained: owns its Medplum query, its loading state, its empty state, its "missing data" badge. This keeps one slow query (e.g., critical labs over a long window) from blocking the rest of the render.
2. **Each card renders a fixed skeleton height.** The initial render shows a 180px-tall Mantine `Skeleton` per card so layout doesn't jump as data lands.
3. **"Missing data" is a first-class state, not an error.** If a query returns an empty result AND the resource *should reasonably exist* for the patient (e.g., no active problems for an admitted patient), the card shows a neutral warning badge `"no data on file"`. If the query errors, it shows `"load failed"` in red. This implements the "Missing-data warnings" target content from the roadmap without depending on clinical-mcp.
4. **No cross-card state.** Cards do not share fetches or context; if two cards read Observations, they issue two queries. The waste is tolerable (≤8 parallel queries on mount) and the isolation is worth it.
5. **Provenance placeholder is a footer line on each card**, not a review affordance. Shows `"source: FHIR · updated {relative time}"` using the newest resource's `meta.lastUpdated`. The actual review/acknowledge affordance is P0.5 (`Review vs acknowledge state model` in matrix), so this is scaffolding only: a stable slot the later feature can populate.
6. **Reuse patterns from P0.1.** `fallback()` helper, the relative-time formatter, and the `data-testid` convention carry over. Overview cards use `data-testid="overview-card-{slug}"` on the root and `data-testid="overview-{slug}-{sub}"` for stable sub-fields (e.g., `overview-problems-count`).

## FHIR query reference

| Card | Query |
|---|---|
| Problems | `Condition?patient=Patient/{id}&clinical-status=active&_sort=-recorded-date` |
| Vitals snapshot (per-vital, 5 queries) | `Observation?patient=Patient/{id}&code=http://loinc.org\|{code}&_sort=-date&_count=1` |
| Critical labs | `Observation?patient=Patient/{id}&category=laboratory&_sort=-date&_count=20` (filtered client-side) |
| Active meds | `MedicationRequest?patient=Patient/{id}&status=active` |
| Recent administrations | `MedicationAdministration?patient=Patient/{id}&effective=ge{now-4h}` |
| Pending tasks | `Task?patient=Patient/{id}&status=requested,accepted,in-progress` |
| Handoff doc | `DocumentReference?subject=Patient/{id}&type=http://loinc.org\|47420-5&doc-status=final&_sort=-date&_count=1` |
| Watch items | `Flag?patient=Patient/{id}&status=active` |
